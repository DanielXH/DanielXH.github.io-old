厚朴前端工程项目规范文档

前言–前端工程诞生的历史

Web前端开发是从网页制作演变而来的，名称上有很明显的时代特征。在互联网的演化进程中，网页制作是Web1.0时代的产物，那时网站的主要内容都是静态的，用户使用网站的行为也以浏览为主。

近年来，随着计算机网络的快速发展，互联网进入Web2.0时代，各种类似桌面软件的Web应用大量涌现，网站的前端由此发生了翻天覆地的变化。网页不再只是承载单一的文字和图片，各种富媒体让网页的内容更加生动，网页上软件化的交互形式为用户提供了更好的使用体验，这些都是基于前端技术实现的。

由计算机网络以及互联网的快速发展带来的前端开发的快速发展，这意味着网页从Web Page逐渐向Web App过渡。现在的前端开发，不仅仅是使用前端三剑客（HTML、CSS、JavaScript）写一个静态页面这么简单，而更多是开发各种Web应用。对于一个复杂的Web应用来说，以一己之力完成所有的开发，显然会有点力不从心；而为了解决一个团队共同开发Web所带来的项目部署、开发流程、维护管理等问题，在此前端工程化诞生了。

前端工程化

前端工程化的意义在于提高效率和降低成本。所谓的效率和成本，不仅是对于团队而言，也是对于个人而言。良好的工程化，能降低沟通成本，实现更好的协同，节省开发和测试人员的重复劳动，降低发布的常见问题等。

不同的开发者对于前端工程话思想理解不同，但其目的都是一样的。在此将前端工程分为四个部分：

  • 1.模块化
  • 2.组件化
  • 3.规范化
  • 4.自动化

1.模块化

模块化就是将一个大文件拆分成互相依赖的小文件，再进行统一的拼装和加载，其目的在于方便多人开发。模块化思想在于尽量隐藏内部作用域，只暴露少量接口供开发者使用，避免多人开发时产生的代码重复与全局污染等问题。

模块化包括两个方面：

  • 1) CSS模块化
  • 2) JS模块化

1) CSS模块化

对于一个页面，如果link一个css文件，http请求数就会增加一个，延长页面加载时间。但为了保持良好的维护性和拓展性，在合理的情况下仍需要拆分css样式表。一个项目的css最基本的结构通常是这样：

  • reset.css：所有页面中的HTML元素默认基础样式和全局样式（如font）；
  • common.css：绝大部分甚至每个页面都会带有的组件，如header、footer等；
  • page.css：页面的具体css样式。

当然一个复杂的项目，其需要拆分的css文件可能不止这么少。

很多css预处理器均可实现css的文件拆分功能，它们是在import/mixin特性支持下实现的。css预处理器有：

  • Sass
  • Less
  • Stylus

在开发过程中，即使按照以上规则，或者使用css预处理器来拆分css样式表，依然没有解决CSS模块化的一个重要问题：选择器的全局污染问题，即仍会产生命名污染的风险。为了解决这个问题，在开发中应制定或采用一定的css命名规范，如BEM(Block-Element-modifier)规则。

2)JS模块化

在ES6之前，JavaScript一直没有模块系统，这对开发大型复杂的前端工程造成了巨大的障碍。对此催生了一些模块加载方案，如CommonJS、AMD和CMD等，某些框架也会有自己模块系统，比如Angular1.x。

  • CMD(SeaJS)
  • AMD(RequireJS)
  • CommonJS(NodeJS)
  • ES6 Module(ECMAScript 2015)

现在ES6已经在语言层面上规定了模块系统，完全可以取代现有的CommonJS和AMD规范，而且使用起来相当简洁，并且有静态加载的特性。

2.组件化

组件化≠模块化。

模块化只是在语言层面上，对代码的拆分；而组件化是基于模块化，在设计层面上，对UI（用户界面）的拆分。

从UI拆分下来的每个包含模板(HTML)+样式(CSS)+逻辑(JS)功能完备的结构单元，称之为组件。

所以除了封装组件本身，还要合理处理组件之间的关系，比如（逻辑）继承、（样式）扩展、（模板）嵌套和包含等，这些关系都可以归为依赖。

目前市面上有很多组件化框架，如以下：

  • Angular
  • React
  • Vue

每个组件化框架各有其优点，也有其缺点，如何选择框架，更多的应考虑实际需求。

可参考vue官方文档的文章：《各组件化框架的优劣性对比》

3.规范化

模块化和组件化确定了开发模型，而规范化则是对开发模式的具体规范进行落实。规范化是前端工程化中很重要的一部分，项目初期规范制定的好坏会直接影响到后期的开发质量，并且应该做了开发前需要首先解决的问题。规范化囊括很多内容，其中上面提到的css命名规范也在此考虑范围之内。规范化包括但不限于：

  • 1.文档：需求文档、设计文档、项目进度管理
  • 2.工具：开发语言选型、开发工具、协同工具、测试工具、发布工具
  • 3.代码规范：html文档规范、css文档规范、JavaScript文档规范、Git分支管理

文档规范涉及到需求分析、设计原型、项目进度管理等方面内容，是一个项目开始的基础以及项目过程的监控。

工具规范是对模块化、组件化、自动化在工具层面上的指导，以开发工具的选型为主，在技术层面和人力层面上影响到一个项目流程的开发效率和开发成本。

代码规范是整个项目正常运行以及后期正常维护的重要保证，上面的工具规范决定了开发者选择什么工具进行开发，而代码规范具体落实到代码如何编写的问题，会涉及到以下多个方面：

  • 命名语义化
  • css、js文件引入方式规定
  • 注释规范
  • 命名规范
  • 代码书写规范(如tab缩进)
  • 字体单位选用规则
  • …

4.自动化

自动化顾名思义，就是让机器帮助我们完成某些工作，让我们不再做机械重复的事情，解放我们的双手。

  • 流程自动化，即人员协同的自动化
  • 代码开发自动化，去掉重复劳动
  • 自动化构建
  • 自动化测试
  • 自动化部署
  • …

如雪碧图的制作，以前是使用PS手动将一张一张图拼起来的，现在使用Gulp+SpriteSmith即可自动完成。

目前的自动化构建工具已经非常丰富，大致可分为几类：

  • 任务管理工具(task runner)：通过声明和组合构建任务来进行整个网站的构建， 有自己的一套任务声明语法和任务实现接口。如Grunt和Gulp，这两个都是插件式的架构，提供大量的插件使用。
  • 打包工具(package tool)：通过为每一类文件配置需要的处理方式，来实现整个站点的构建。如 Webpack 和 FIS ，这两个都是整个站点的整体构建解决方案。

任务管理工具中我们可以声明若干个任务，比如压缩、测试、替换等。这些任务间可互相依赖（往往用来定义顺序），可以是同步的也可以是异步的。 然后我们可以自由地选择运行哪个任务，任务管理工具会帮我们运行它（以及它的依赖）。

Webpack给出了一体化的构建方案。我们无需关心构建过程的实现细节，只需要为每种文件设置它应该经过哪些预处理、哪些转换，并且给定打包地址。Webpack甚至会自动替换HTML中的相应资源。
